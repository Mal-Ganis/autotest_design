简介
AutoTestDesign AI 应用是一个被提议的系统，旨在自动化和增强软件测试设计活动，符合行业标准，如 ISTQB 基础级原则以及 ISO/IEC/IEEE 29119-4 中定义的详细测试技术。其主要目标是利用人工智能（AI）和机器学习（ML）进行需求分析、风险评估和系统化测试用例生成，从而提升测试效率和覆盖范围。

目标与宗旨
G1：标准对齐：确保生成的测试工件（测试用例、优先级评分）符合既定测试标准（ISTQB术语和ISO 29119结构）。
G2：自动化：自动化复杂的基于规则的测试设计技术（如EP、BVA、决策表）。
G3：基于风险的测试：在流程早期整合风险分析，以优先安排测试工作。
G4：模块化：设计一个平台，方便将新的测试设计算法和AI模型作为独立模块轻松集成。

专题类别

描述

FR 1.0

输入/解析

系统必须能够从各种来源（例如 CSV、纯文本或直接用户输入）导入软件需求。

FR 1.1

需求结构

系统必须使用自然语言处理（NLP）来解析和标记原始文本，识别输入字段、数据范围、条件和预期动作等关键组成部分。

FR 2.0

风险分析与优先级

系统必须采用AI/ML模型（项目1）为每个导入需求分配风险评分和测试优先级（高、中、低）。

FR 3.0

黑盒测试设计

系统必须包含模块，自动应用并生成至少三个来自ISO 29119-4核心黑箱技术的测试用例（例如：等价划分、边界值分析、决策表）。

FR 4.0

白盒测试建模

系统必须包含一个模块，用于建模系统行为（例如状态转换图），并生成针对所选覆盖标准的最优测试序列（例如所有状态）。

FR 5.0

测试神谕生成

系统必须包含一个组件（项目10），以帮助综合特定需求和特定测试数据的预期结果。

FR 6.0

输出与出口

系统必须生成结构化、标准格式（如 JSON 或 Excel/CSV）的测试产物（测试用例、测试套件、风险评分），以便导入测试管理工具。

FR 7.0

测试套件优化

系统必须包含优化模块（项目4或9），以根据风险或覆盖效率优先级或最小化生成的测试套件。

非功能需求（NFR）
4.1. 性能

NFR 4.1.1（分析时间）：对100项需求批量的需求处理和风险分析必须在5秒内完成。
NFR 4.1.2（测试用例生成）：为单一需求（例如决策表）生成完整测试用例集的时间不得超过2秒。
4.2. 可用性（用户体验/界面）

NFR 4.2.1（接口）：应用程序必须提供一个简洁、直观的用户界面（UI），以便输入需求并查看生成的测试成果。
NFR 4.2.2（可追溯性）：所有生成的测试用例必须明确关联到原始需求ID和所用设计技术。
4.3. 安全性

NFR 4.3.1（数据处理）：如果需求数据包含敏感信息，系统必须实现基本的访问控制（不过对于学生项目，这可能会简化）。
4.4. 可维护性与技术

NFR 4.4.1（技术栈）：应用应使用适合AI/ML集成的现代开源技术栈开发（例如后端逻辑用Python/Flask/Django，UI用HTML/CSS/JavaScript或前端框架）。
NFR 4.4.2（模块化）：架构必须强制核心框架与单个测试设计模块（项目1-10）之间实现松耦合，以便算法的独立开发和交换。
NFR 4.4.3（文档）：所有代码必须有充分的文档记录，项目还必须包含系统架构图和用户手册。
技术限制
C 5.1（人工智能框架）：学生必须使用常用的AI/ML库（例如Scikit-learn、TensorFlow、PyTorch或用于自然语言处理的spaCy）来学习各自的模块。
C 5.2（数据）：系统必须包含需求和历史执行结果的示例模拟数据集，用于训练和演示目的。
C 5.3（语言）：所有核心逻辑和用户界面必须用所选的主要开发语言（例如 Python）开发。
