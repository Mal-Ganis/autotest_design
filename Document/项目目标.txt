## 一、我们要做什么事？

这个项目本质上分为**两条线**：

### 线1：开发一个 AI 驱动的测试设计工具（AutoTestDesign）
你要做一个能自动分析需求、评估风险、生成测试用例的软件工具。这个工具必须至少实现以下核心功能：

| 功能 | 说明 |
|------|------|
| **需求导入** (FR 1.0) | 支持 CSV、纯文本、用户输入等方式导入软件需求 |
| **需求解析** (FR 1.1) | 用 NLP/规则解析需求文本，提取输入字段、数据范围、条件、预期动作 |
| **风险分析** (FR 2.0) | 给每个需求打风险分，分高/中/低优先级 |
| **黑盒测试生成** (FR 3.0) | 自动生成至少 3 种测试用例：等价类划分、边界值分析、决策表 |
| **结果导出** (FR 6.0) | 输出 JSON / Excel / CSV，方便导入其他测试管理工具 |
| **交互式审查** | 允许测试人员（"设计师"）在工具里手动修改覆盖项、测试策略、测试用例 |

**可选加分功能**（做了能拿额外学分）：
- 白盒测试建模（FR 4.0）：画状态转换图，生成覆盖所有状态的测试序列
- 测试预言生成（FR 5.0）：自动预测预期结果
- 测试套件优化（FR 7.0）：基于风险或覆盖率对测试集排序/精简

### 线2：用这个工具去测试一个"目标应用"
你需要**另外选一个简单的真实应用**（比如：登录系统、计算器、TodoList 网页应用），然后用你开发的工具来测试它，产出以下文档：

| 交付物 | 权重 | 内容 |
|--------|------|------|
| **风险分析报告** | 10% | 分析目标应用各功能的风险 |
| **测试计划** | 40% | 项目范围、测试项、高层测试设计、进度表、团队分工、测试框架选型理由、成本估算（与手工测试对比） |
| **详细测试设计与执行文档** | 30% | 用工具设计测试用例、解释覆盖率、多种黑盒+白盒技术、测试脚本实现、执行结果分析 |

## 二、具体怎么做？（建议执行路线）

### 阶段1：确定目标应用（第1-2天）
选一个**简单但有足够测试点**的目标应用，推荐：
- **登录注册模块**（有输入验证、边界值、状态转换、权限控制，测试点丰富）
- **计算器**（适合等价类、边界值、决策表）
- **TodoList 网页应用**（有增删改查、状态变化）

**建议选登录系统**，因为它天然包含：
- 输入字段（用户名、密码）→ 适合等价类、边界值
- 业务规则（密码强度、锁定策略）→ 适合决策表
- 状态变化（未登录→已登录→锁定）→ 适合状态转换图（白盒）

### 阶段2：搭建工具框架（第3-7天）
**技术栈建议**（符合文档要求的"现代开源技术栈"）：
- **后端**：Python + Flask/FastAPI（轻量，适合学生项目）
- **AI/NLP**：spaCy / 正则规则 / 简单 ML（Scikit-learn）做需求解析和风险评分
- **前端**：HTML + JavaScript 或 React（做交互式审查界面）
- **数据存储**：SQLite / JSON 文件（简单够用）

**工具架构设计要点**：
```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   前端 UI   │────▶│   后端 API  │────▶│  核心引擎   │
│ (输入/审查) │     │ (Flask)     │     │(解析/分析/生成)│
└─────────────┘     └─────────────┘     └──────┬──────┘
                                               │
                    ┌─────────────┬────────────┼────────────┐
                    ▼             ▼            ▼            ▼
               需求解析模块   风险分析模块  黑盒生成模块  导出模块
               (FR 1.1)      (FR 2.0)     (FR 3.0)     (FR 6.0)
```

**核心实现思路**：
1. **需求解析**：不必做复杂 NLP，先用规则引擎 + 关键词匹配提取"输入/条件/动作"
2. **风险评分**：基于规则（如"涉及支付"=高风险，"仅展示"=低风险）或简单决策树模型
3. **黑盒生成**：
   - 等价类划分：根据数据类型和范围自动分有效/无效等价类
   - 边界值分析：在等价类边界 ±1 处取值
   - 决策表：提取条件组合，生成规则表
4. **交互式审查**：前端表格展示生成的测试用例，允许用户增删改、调整优先级

### 阶段3：测试目标应用（第8-12天）
1. **用工具分析目标应用的需求**：把登录系统的需求文本输入工具
2. **生成测试用例**：导出工具生成的测试用例（等价类、边界值、决策表）
3. **补充白盒测试**：手动设计状态转换图，补充状态覆盖用例
4. **执行测试**：
   - 选测试框架：Selenium（Web 自动化）或 PyTest（单元/接口测试）
   - 写自动化脚本执行生成的测试用例
   - 记录结果（通过/失败/缺陷）

### 阶段4：撰写文档与制作演示（第13-14天）
按权重优先级，重点写好**测试计划（40%）**和**详细测试设计（30%）**。

**测试计划必须包含**：
- 项目范围与目标
- 目标应用的功能/非功能特性描述 + 架构图
- 高层测试套件设计：每个功能用什么测试技术（如"登录功能用等价类+边界值"）
- 进度表（里程碑）
- 组织架构图（谁负责什么）
- 测试框架选型理由（为什么选 Selenium/PyTest）
- **成本估算**：估算使用工具 vs 手工测试节省的人时（这是评分点！）

**详细测试设计必须体现**：
- 概念 → 覆盖项识别 → 覆盖策略 → 测试用例 → 提示设计 → 结果分析 → 基于证据改进
- 工具生成的原始用例 + 你作为"设计师"人工审查修改后的最终用例
- 测试用例与需求 ID、覆盖项的追溯关系
- 实际执行结果和缺陷分析

### 阶段5：打包提交（第15天）
提交清单：
- [ ] 工具源码 + README（安装/运行说明）+ 视频演示（压缩包）
- [ ] 风险分析报告（PDF）
- [ ] 测试计划（PDF）
- [ ] 详细测试设计与执行文档（PDF）
- [ ] 演示 PPT（第一张放团队 ID、姓名、学号）

## 三、关键提醒（避免扣分）

1. **工具 ≠ 目标应用**：所有文档（风险报告、测试计划、详细设计）都是关于"被测系统"（如登录模块），不是写你开发的 AutoTestDesign 工具本身。
2. **交互式审查是硬性要求**：工具不能只"一键生成"，必须提供界面让测试人员能改覆盖项、策略、用例。这是"主要内容"段落强调的。
3. **性能要求要努力满足**：100 条需求处理 < 5 秒，单需求生成 < 2 秒。如果做不到，文档里必须写详细分析和改进建议，否则扣分。
4. **成本估算必须做**：在测试计划里估算使用工具测试目标应用的成本（人时、资源），并与纯手动测试做对比。
5. **可追溯性**：每个测试用例必须关联到原始需求 ID 和使用的技术（等价类/BVA/决策表），这是 NFR 要求。
