# 详细测试设计与执行文档

## 1. 引言

### 1.1 文档目标

本文件面向 AutoTestDesign 项目的整体测试设计交付，重点说明黑盒测试设计模块如何把已经完成的结构化需求、风险优先级和覆盖项，进一步转化为可执行的测试设计成果。与单纯列出测试用例不同，本项目的详细测试设计目标是建立一条稳定的设计链路：首先解释所采用的测试设计概念，然后识别每条需求对应的覆盖项，再为覆盖项选择合适的黑盒测试技术，最后生成带有需求 ID、覆盖项 ID 和策略 ID 的测试用例。

从工程角度看，本测试设计工作的价值不只在于“生成了多少条用例”，更在于每一条用例都能说明自己为什么存在、覆盖了什么需求、对应哪一个覆盖项、由哪一种测试技术生成，以及后续执行结果如何支撑当前测试设计的有效性。因此，本文档采用“概念 → 覆盖项识别 → 覆盖策略 → 测试用例 → 提示设计 → 结果分析 → 基于证据改进”的顺序展开，使读者能够清楚看到项目从输入产物到最终测试设计成果的完整推导过程。

### 1.2 被测对象概述

`target-login-app` 是一个用于课程项目验证的本地 Web 登录系统，提供用户注册、用户登录、账户锁定、会话访问、个人资料查看等核心功能。该应用使用 Flask 提供后端接口，使用 SQLite 保存用户、密码哈希和登录状态相关信息，前端页面通过静态 HTML 与接口交互。

| 维度             | 说明                                                                                                                        |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------- |
| 应用名称         | `target-login-app`                                                                                                          |
| 技术栈           | Python Flask、Flask-SQLAlchemy、SQLite、bcrypt、HTML/CSS/JavaScript                                                         |
| 本次测试涉及接口 | `/api/register`、`/api/login`、`/api/logout`、`/api/profile`；应用另提供密码重置相关接口 `/api/reset-request`、`/api/reset` |
| 测试范围         | 用户名长度边界、登录失败锁定、锁定期间访问控制、登录后资料访问、登出后会话控制                                              |
| 测试执行方式     | 使用 Flask `test_client` 进行接口级验证，并结合响应状态码与业务 code 判断执行结果                                           |

## 2. 测试设计概念

### 2.1 等价类划分

等价类划分是一种将输入域划分为若干有效类和无效类的黑盒测试技术。它的核心思想是：同一类中的输入值应当触发系统相同或相近的处理逻辑，因此可以选取代表值进行测试，以较少数量的用例覆盖较大的输入空间。在本项目中，用户名长度 `[3, 20]` 就是典型的等价类划分对象。长度位于区间内部的用户名属于有效类；长度小于 3、长度大于 20、空输入等属于无效类。

等价类划分在本项目中主要用于两个方面。第一，用于验证输入字段本身的合法性，例如用户名是否满足长度约束。第二，用于验证系统动作是否在不同输入类别下正确触发，例如账户锁定动作在触发条件满足时发生，在触发条件不满足时不发生。通过这种方式，测试设计流程能够把自然语言需求转化为结构化、可执行的测试场景。

### 2.2 边界值分析

边界值分析基于一个常见测试经验：问题风险往往更容易出现在输入区间的边界附近，而不是区间中间。因此，对于闭区间 `[3, 20]`，测试设计不仅覆盖区间内部值，还会测试边界值及其相邻值。当前生成策略使用六点边界法，选择 `2、3、4、19、20、21`，分别覆盖下边界外、下边界、下边界内邻近值、上边界内邻近值、上边界和上边界外。

在登录系统中，边界值分析具有明确意义。用户名过短可能无法提供足够的账户区分度，用户名过长可能影响数据库字段、页面展示或接口处理。通过边界测试，可以确认系统对临界输入的处理稳定、清晰且符合需求预期。

### 2.3 决策表测试

决策表测试适用于系统行为由条件组合决定的场景。FR-LOGIN-002 中的“密码连续错误超过 5 次”就是一个条件分支需求。当条件不满足时，系统应保持普通登录失败处理；当条件满足时，系统应进入账户锁定状态，并限制后续登录。

本测试设计使用决策表技术将条件分支显式展开。即使当前需求只有一个条件，仍然可以形成两条规则：条件为 False 时不触发锁定，条件为 True 时触发锁定。这样的表示方式让测试设计更加清晰，也便于后续扩展更多条件，例如用户是否存在、账户是否已锁定、锁定时间是否过期等。

### 2.4 提示设计

本项目中的提示设计不是简单的自然语言说明，而是作为策略与用例生成之间的桥梁存在。`strategies_and_prompts.py` 会为每条策略生成 `prompt_notes` 字段，记录该策略应关注的输入范围、边界值、条件桩、动作桩或异常组合。这样做有两个好处：一是增强生成过程的可解释性，测试设计人员能够看到工具为什么生成某些用例；二是为后续接入 AI 生成器或人工审查界面提供稳定上下文。

例如，针对用户名长度需求，提示中会记录有效区间 `[3, 20]` 和边界值 `2, 3, 4, 19, 20, 21`。针对账户锁定需求，提示中会记录条件桩“密码连续错误超过 5 次”和动作桩“锁定账户 30 分钟”。这些提示使测试用例不是孤立生成，而是从覆盖意图和测试技术中自然推导出来。

## 3. 覆盖项识别

### 3.1 覆盖项识别方法

覆盖项是从需求到测试用例之间的中间层。需求通常是面向业务或用户行为的自然语言描述，而测试用例需要具体到输入、步骤和预期结果。覆盖项的作用就是把需求拆解为可测试的验证目标，使后续策略分配和用例生成具备明确依据。

当前覆盖项由风险与覆盖分析流程生成，黑盒测试设计模块在此基础上进行消费。该模块不改变覆盖项的语义，而是读取覆盖项中的 `coverage_id`、`linked_req_ids`、`focus` 和 `notes`，判断它适合采用哪一种黑盒技术。这样既保持了各流水线步骤之间的数据契约，也保证了从需求到用例的追溯关系不会断裂。

### 3.2 覆盖项清单

| 覆盖项 ID | 关联需求     | Focus 类型           | 覆盖意图                                                                   |
| --------- | ------------ | -------------------- | -------------------------------------------------------------------------- |
| COV-001   | FR-LOGIN-001 | `data_range`         | 验证用户名长度区间与边界，覆盖有效输入、过短输入、过长输入和空输入等场景。 |
| COV-002   | FR-LOGIN-002 | `condition_branch`   | 覆盖密码连续错误次数达到阈值前后的系统分支行为。                           |
| COV-003   | FR-LOGIN-002 | `expected_action`    | 验证账户锁定动作是否在条件满足时正确发生。                                 |
| COV-004   | FR-LOGIN-002 | `expected_action`    | 验证锁定后系统是否限制登录或拒绝继续访问。                                 |
| COV-005   | FR-LOGIN-003 | `general_functional` | 验证用户成功登录后能够访问个人资料页或资料接口。                           |
| COV-006   | FR-LOGIN-003 | `risk_negative`      | 补充未登录、登出后、无有效会话等高风险负面访问场景。                       |

### 3.3 覆盖项质量说明

本次覆盖项能够覆盖三条需求的主要验证目标。FR-LOGIN-001 被识别为数据范围类覆盖项，适合通过等价类划分和边界值分析进行细粒度验证。FR-LOGIN-002 被拆分为条件分支和预期动作两个层次，既覆盖了“什么时候锁定”，也覆盖了“锁定后产生什么效果”。FR-LOGIN-003 虽然结构化信息较少，但通过通用功能覆盖和高风险负面覆盖，仍然形成了登录后访问成功、无会话访问失败、登出后访问失败等关键场景。

从测试设计角度看，这种覆盖项拆分方式是合理的。它没有把需求直接压缩成少数端到端用例，而是保留了验证目标之间的差异，使后续策略分配更加清晰。覆盖项越清晰，生成用例时越容易说明每条用例的存在价值，也越容易在执行后进行结果分析和覆盖评价。

## 4. 覆盖策略

### 4.1 覆盖策略生成规则

`strategies_and_prompts.py` 根据覆盖项的 `focus` 类型自动选择黑盒测试技术。策略生成遵循固定映射规则，确保同类覆盖项使用稳定、一致、可解释的方法。

| Focus 类型           | 分配技术 | 设计依据                                              |
| -------------------- | -------- | ----------------------------------------------------- |
| `data_range`         | EP、BVA  | 数据范围需要同时覆盖有效/无效输入类别和边界附近取值。 |
| `condition_branch`   | DT       | 条件分支需要明确列出条件真假组合及对应动作。          |
| `expected_action`    | EP       | 动作验证可以划分为触发动作和不触发动作两个类别。      |
| `general_functional` | EP       | 通用功能路径适合先覆盖有效路径和无效路径。            |
| `risk_negative`      | BVA、DT  | 高风险场景需要补充边界、异常条件和非法组合。          |

### 4.2 策略清单

| 策略 ID | 技术 | 关联覆盖项 | 关联需求     | 风险优先级 | 提示要点                                         |
| ------- | ---- | ---------- | ------------ | ---------- | ------------------------------------------------ |
| STR-001 | EP   | COV-001    | FR-LOGIN-001 | Low        | 对用户名划分有效区间、过短、过长、空输入等类别。 |
| STR-002 | BVA  | COV-001    | FR-LOGIN-001 | Low        | 选择用户名长度边界值 `2、3、4、19、20、21`。     |
| STR-003 | DT   | COV-002    | FR-LOGIN-002 | Medium     | 构建“连续错误超过 5 次”条件真假组合。            |
| STR-004 | EP   | COV-003    | FR-LOGIN-002 | Medium     | 验证锁定账户 30 分钟动作的触发与不触发。         |
| STR-005 | EP   | COV-004    | FR-LOGIN-002 | Medium     | 验证限制登录或锁定账户动作是否生效。             |
| STR-006 | EP   | COV-005    | FR-LOGIN-003 | High       | 覆盖登录后访问资料页的有效路径与异常路径。       |
| STR-007 | BVA  | COV-006    | FR-LOGIN-003 | High       | 对高风险场景补充边界或临界状态检查。             |
| STR-008 | DT   | COV-006    | FR-LOGIN-003 | High       | 对会话存在与否、访问动作是否允许构建规则组合。   |

### 4.3 策略分布分析

本次共生成 8 条策略，覆盖 EP、BVA、DT 三类黑盒技术。其中 EP 策略 4 条，主要用于输入分类和动作触发验证；BVA 策略 2 条，主要用于用户名长度边界和高风险边界场景；DT 策略 2 条，主要用于账户锁定条件和会话访问规则组合。该分布与需求特征相匹配：输入约束需要 EP 和 BVA，条件分支需要 DT，登录后访问路径需要功能类与负面类场景共同覆盖。

## 5. 测试用例设计

测试用例设计阶段读取 `06_test_cases_draft.json` 中的生成结果，并按照黑盒测试技术类型进行组织。为了让测试设计过程更清晰，本节不只给出用例数量，还进一步说明每组用例的来源策略、关联需求、覆盖项和设计意图。这样可以保证读者不仅看到“生成了哪些用例”，也能理解这些用例如何从覆盖策略自然推导出来。

本次共生成 25 条测试用例，其中 EP 用例 14 条、BVA 用例 7 条、DT 用例 4 条。三类用例共同覆盖用户名输入合法性、账户锁定条件、锁定动作验证、登录后资料访问和会话相关负面场景。

### 5.1 EP 用例（14 条）

等价类划分用例主要用于验证有效输入与无效输入类别。对于有明确数据范围的字段，系统会生成有效类、下溢无效类、上溢无效类和空输入无效类；对于动作验证类覆盖项，系统会生成“条件满足时触发动作”和“条件不满足时不触发动作”两类用例；对于通用功能需求，系统会生成有效路径与无效路径的代表用例。

#### 5.1.1 用户名等价类（STR-001 → TC-001 ~ TC-004）

基于 FR-LOGIN-001 的数据范围 `[3, 20] chars`，用户名输入被划分为一个有效类和三个无效类。有效类选择区间内代表值，验证系统能够接受合法长度；无效类分别覆盖下溢、上溢和空输入，验证系统能够拒绝不符合约束的输入。

| 用例 ID | 类别               | 测试数据          | 预期结果                 |
| ------- | ------------------ | ----------------- | ------------------------ |
| TC-001  | 有效类（区间中值） | 用户名 = 11 chars | 系统接受，处理成功       |
| TC-002  | 无效类（下溢）     | 用户名 = 2 chars  | 系统拒绝，提示输入不合法 |
| TC-003  | 无效类（上溢）     | 用户名 = 21 chars | 系统拒绝，提示输入不合法 |
| TC-004  | 无效类（空输入）   | 用户名 = `""`     | 系统拒绝，提示必填字段   |

**设计说明**：有效类选取区间中值 `(3 + 20) // 2 = 11`，该值处于合法区间内部，适合作为普通有效输入代表。无效类分别取 `min - 1 = 2` 和 `max + 1 = 21`，用于验证系统对越界长度的拒绝能力；空输入用例用于覆盖用户未填写字段的缺省场景。该组用例与 FR-LOGIN-001、COV-001 和 STR-001 建立直接追溯关系。

#### 5.1.2 账户锁定动作验证（STR-004 → TC-013 ~ TC-016）

基于 FR-LOGIN-002 的预期动作，系统需要在密码连续错误达到阈值后锁定账户，并在条件不满足时保持正常流程。该组用例从动作是否触发的角度验证锁定行为，重点不是输入字段本身，而是系统在不同触发条件下是否执行正确动作。

| 用例 ID | 场景                                     | 触发条件   | 预期结果                 |
| ------- | ---------------------------------------- | ---------- | ------------------------ |
| TC-013  | 锁定账户 30 分钟（正常触发）             | 条件满足   | 动作成功触发             |
| TC-014  | 锁定账户 30 分钟（条件不满足时不触发）   | 条件不满足 | 动作未触发，系统正常继续 |
| TC-015  | 限制登录或锁定账户（正常触发）           | 条件满足   | 动作成功触发             |
| TC-016  | 限制登录或锁定账户（条件不满足时不触发） | 条件不满足 | 动作未触发，系统正常继续 |

**设计说明**：该组用例面向 COV-003，关注“锁定账户 30 分钟”这一动作是否正确发生。对于状态类需求，仅验证错误消息是不充分的，还需要确认系统状态发生了预期变化。因此，这组用例在设计上为后续执行阶段保留了明确观察点：当触发条件满足时，账户应进入锁定状态；当触发条件不满足时，不应产生锁定副作用。

#### 5.1.3 限制登录动作验证（STR-005 → TC-017 ~ TC-020）

与 STR-004 类似，STR-005 通过 COV-004 覆盖项独立验证“限制登录或锁定账户”的动作效果。该覆盖项强调锁定之后对登录行为的限制，确保系统不仅能够写入锁定状态，也能够在后续登录请求中正确执行访问控制。

| 用例 ID | 场景                           | 预期结果     |
| ------- | ------------------------------ | ------------ |
| TC-017  | 锁定账户 30 分钟（正常触发）   | 动作成功触发 |
| TC-018  | 锁定账户 30 分钟（不触发）     | 动作未触发   |
| TC-019  | 限制登录或锁定账户（正常触发） | 动作成功触发 |
| TC-020  | 限制登录或锁定账户（不触发）   | 动作未触发   |

**设计说明**：COV-003 与 COV-004 都与账户锁定相关，但关注点不同。COV-003 更关注锁定动作是否产生，COV-004 更关注锁定后的登录限制是否生效。将两者分别建模，有助于在追溯矩阵中清楚说明“状态写入”和“状态生效”这两个验证层次。

#### 5.1.4 通用功能验证（STR-006 → TC-021 ~ TC-022）

基于 FR-LOGIN-003 的登录跳转与会话访问需求，系统生成一组通用功能验证用例。由于该需求在结构化结果中没有显式数据范围和条件分支，因此采用有效输入与无效输入的代表路径进行覆盖。

| TC-022  | 无效输入 | 系统拒绝无效输入并给出明确提示 |

**设计说明**：该组用例对应 COV-005，重点验证用户成功登录后的目标行为是否成立，以及异常路径下系统是否能给出合理处理。对于登录后资料访问这类功能，正向路径能够证明主流程可用，反向路径则能够证明系统不会在无效输入或无效状态下错误放行。

### 5.2 BVA 用例（7 条）

#### 5.2.1 用户名边界值（STR-002 → TC-005 ~ TC-010）

| 用例 ID | 边界点   |   值 | 有效性 | 预期结果                   |
| ------- | -------- | ---: | ------ | -------------------------- |
| TC-005  | 下边界-1 |    2 | 无效   | 系统拒绝，提示输入超出范围 |
| TC-006  | 下边界   |    3 | 有效   | 系统接受，处理成功         |
| TC-007  | 下边界+1 |    4 | 有效   | 系统接受，处理成功         |
| TC-008  | 上边界-1 |   19 | 有效   | 系统接受，处理成功         |
| TC-009  | 上边界   |   20 | 有效   | 系统接受，处理成功         |
| TC-010  | 上边界+1 |   21 | 无效   | 系统拒绝，提示输入超出范围 |

**设计说明**：采用三值边界分析思想，即围绕每个边界取边界值本身及其相邻值。闭区间 `[3, 20]` 下，`min=3` 和 `max=20` 为有效边界值，`2` 和 `21` 为无效越界值。该组用例能够验证系统是否正确处理最小合法长度、最大合法长度以及刚好越界的输入。

#### 5.3.1 账户锁定决策表（STR-003 → TC-011 ~ TC-012）

## 9. 结果分析

### 9.1 覆盖完整性分析

本次详细测试设计覆盖了 3 条输入需求、6 个覆盖项、8 条测试策略和 25 条工具原始用例。经过执行用例整理后，形成 8 条接口级执行用例，并全部保留需求、覆盖项和策略追溯信息。该结果说明测试设计产物满足“从需求到用例可追溯”的基本要求，也满足“至少使用三种黑盒测试技术”的设计目标。

| 统计维度         |         数量 |
| ---------------- | -----------: |
| 功能需求         |         3 条 |
| 覆盖项           |         6 项 |
| 策略             |         8 条 |
| 工具原始用例     |        25 条 |
| EP 用例          | 14 条（56%） |
| BVA 用例         |  7 条（28%） |

| 接口级执行用例   |         8 条 |

从分布上看，FR-LOGIN-001 和 FR-LOGIN-002 各形成 10 条原始用例，分别占 40%；FR-LOGIN-003 形成 5 条原始用例，占 20%。这种分布与需求结构相符：FR-LOGIN-001 具备明确长度范围，因此可以展开较多 EP 与 BVA 用例；FR-LOGIN-002 同时包含条件分支和预期动作，因此需要 DT 与 EP 共同覆盖；FR-LOGIN-003 风险优先级最高，但结构化字段较少，因此采用通用功能覆盖与高风险负面场景补充的方式保证主路径和异常路径均被纳入设计。

从执行结果看，接口级执行用例覆盖的主要功能路径均达到预期。用户名边界验证能够区分有效边界和无效边界，说明输入校验规则在长度维度上表现稳定。账户锁定相关用例验证了从错误密码累积到锁定状态，再到锁定期间拒绝登录的完整链路，说明系统能够根据连续失败条件改变账户状态。资料访问相关用例验证了登录状态与会话访问控制之间的关系，登录后可以访问资料，登出后无法继续访问受保护接口。

## 10. 用例生成机制说明

### 10.1 EP 生成器逻辑

EP 生成器首先读取策略中的 `linked_req_ids` 和 `linked_coverage_ids`，再从需求中提取 `data_ranges`、`conditions` 和 `expected_actions`。当需求包含明确数据范围时，生成器会为每个范围生成有效类、下溢无效类、上溢无效类和空输入无效类；当覆盖项属于动作验证时，生成器会围绕动作触发与不触发展开用例；当结构化信息较少时，生成器会退回到通用功能验证路径。

```python
输入: strategy（含 linked_req_ids, linked_coverage_ids）

IF 关联需求含 data_ranges:
    FOR 每个 range:
        生成有效类: 区间中值 mid = lo + (hi - lo) // 2
        生成无效类-下溢: lo - 1
        生成无效类-上溢: hi + 1
        生成无效类-空输入: ""

ELIF 覆盖项 focus == expected_action:
    FOR 每个 action:
        生成正常触发用例
        生成不触发用例

ELSE:
    生成通用有效输入用例
    生成通用无效输入用例
```

该逻辑对应当前 EP 用例数量：STR-001 生成 4 条用户名等价类用例，STR-004 和 STR-005 各生成 4 条动作验证用例，STR-006 生成 2 条通用功能用例，总计 14 条。

### 10.2 BVA 生成器逻辑

BVA 生成器优先查找需求中的数值或长度范围。当范围存在且上下界为整数时，生成器采用六点边界法；当范围不存在时，生成器生成通用边界探测用例，保留对高风险边界状态的测试设计入口。

```python
输入: strategy（含 linked_req_ids）

IF 关联需求含 data_ranges:
    FOR 每个整数闭区间 range:
        生成 6 个测试点:
        (lo-1, 无效), (lo, 有效), (lo+1, 有效)
        (hi-1, 有效), (hi, 有效), (hi+1, 无效)

ELSE:
    生成通用边界探测用例
```

该逻辑对应当前 BVA 用例数量：STR-002 根据用户名长度范围生成 6 条边界用例，STR-007 在缺少明确数值范围时生成 1 条通用边界探测用例，总计 7 条。

### 10.3 DT 生成器逻辑

DT 生成器从关联需求中收集 `conditions` 和 `expected_actions`。如果需求中没有显式条件，生成器使用“前置条件成立”作为默认条件桩；如果没有显式动作，则使用“系统执行预期动作”作为默认动作桩。随后按条件数量展开 `2^n` 条组合规则。

```python
输入: strategy（含 linked_req_ids）

收集所有 conditions 和 expected_actions
IF 无 conditions: 默认 ["前置条件成立"]
IF 无 actions:    默认 ["系统执行预期动作"]

n_combos = 2 ^ len(conditions)
FOR combo_idx in range(n_combos):
    计算每个 condition 的 True/False 值
    IF 所有条件为 True:
        expected = 所有动作连接
    ELSE:
        expected = 条件未全部满足，动作不触发或走替代分支
```

该逻辑对应当前 DT 用例数量：STR-003 对“密码连续错误超过 5 次”生成 2 条规则；STR-008 对默认前置条件生成 2 条规则，总计 4 条。
