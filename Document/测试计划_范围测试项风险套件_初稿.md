## 一、项目范围（Project Scope）

### 1.1 背景（Background）

课程期末项目要求小组在开发 **AI 驱动的 AutoTestDesign 工具** 的同时，选取**独立于工具**的目标应用，用工具辅助完成需求分析、风险评估与系统化测试设计，并提交测试计划、风险分析报告等交付物（*Assignment 2* §1.2）。

本测试计划针对的**被测系统**为 **target-login-app**：基于 **Python Flask** 与 **SQLite** 的 Web 登录演示系统，提供注册、登录与会话、个人资料、连续失败锁定、密码重置等能力；前端为 **静态 HTML/CSS/JS**，通过 **JSON API** 与后端交互，统一响应格式为 `{ code, message, data }`（见 `target-login-app/README.md` 与 `app.py`）。

### 1.2 测试活动总体目标（Overall Objectives）

| 目标编号 | 目标描述 | 与课程要求的对应 |
|----------|----------|------------------|
| O-1 | **验证功能正确性**：注册/登录/锁定/会话/资料/登出/重置等路径在约定输入下行为与 `README` 及当前实现一致。 | 支撑作业「用工具测目标应用」的**可演示、可追溯**结论。 |
| O-2 | **暴露与风险登记一致的高风险行为**：尤其是会话与密钥（FR 1.1、NFR 1.1.2）、锁定与「用户名不存在」路径（FR 1.2）、演示令牌与枚举线索（FR 1.5）。 | 与 *Risk Analysis Report* 及工具 **FR 2.0** 优先级分配一致。 |
| O-3 | **覆盖主要非功能关切（NFR 1.1.2）**：安全相关配置（如 `LOGIN_APP_SECRET`）、Cookie 属性、错误响应不泄露堆栈、演示与生产边界说明。 | 对齐 *Assignment 2* §1.1.2 **Security** 等 NFR 条目。 |
| O-4 | **为《详细测试设计与执行文档》预留输入**：在某一主功能模块上形成可执行的用例集与追溯关系（后续文档撰写）。 | 对应 *Assignment 2* §1.2 第 4 项。 |

### 1.3 测试范围（In Scope）

- **功能范围**：`POST /api/register`、`/api/login`、`/api/logout`、`GET /api/profile`、`POST /api/reset-request`、`POST /api/reset`；静态入口页及 `login.html`、`register.html`、`profile.html`、`reset.html` 与上述 API 的联调路径。  
- **数据与规则**：`services.py` 中用户名/密码/邮箱校验规则；`MAX_FAILED=5`、`LOCK_MINUTES=30` 锁定逻辑；`PasswordResetToken` 有效期与 `used` 标记（见 `models.py`、`app.py`）。  
- **环境与配置**：本地 `http://127.0.0.1:5000`（或团队约定端口）；可选对比「已设置 `LOGIN_APP_SECRET`」与「未设置（默认密钥）」两种配置下的会话行为（用于安全说明，非修改产品代码的必测项）。

### 1.4 不在范围（Out of Scope，初稿约定）

- **AutoTestDesign 工具**内部流水线脚本自身的单元测试（由工具组内分工另述）。  
- **真实邮件投递**、**第三方支付**、**OAuth 第三方登录**：当前 `target-login-app` **未实现**（`README` 与代码无对应模块），不测。  
- **大规模性能基准与渗透测试**：课程演示环境以**手工 + 轻量自动化**为主；若需数据须另列资源与计划（完整测试计划其他小节可补充）。

## 二、测试项（Test Items）

### 2.1 目标应用主要功能特性（Functional）

| 测试项 ID | 功能模块 | 主要行为 / 接口 | 说明与追溯 |
|-----------|----------|-----------------|------------|
| TI-F-01 | 用户注册 | `POST /api/register` | 用户名唯一性、格式（3–20，字母数字下划线）、密码复杂度、确认密码、邮箱格式；错误码 1001–1005。 |
| TI-F-02 | 用户登录 | `POST /api/login` | 成功建立会话；空字段 2001；错误凭据 2002/401；锁定 2003/403。 |
| TI-F-03 | 失败计数与锁定 | 登录失败路径 + `users` 表字段 | 与 **FR 1.2** 高风险一致：已存在用户错误密码累计至 5 次后锁定 30 分钟；**用户名不存在时不累计失败次数**（代码行为，用例需记录）。 |
| TI-F-04 | 会话与个人资料 | `session`、`GET /api/profile` | 未登录 3001；成功返回 `to_public_dict()`（含 `email`，不含密码哈希）。 |
| TI-F-05 | 登出 | `POST /api/logout` | 会话清除后 `GET /api/profile` 应未授权。 |
| TI-F-06 | 密码重置请求 | `POST /api/reset-request` | 演示返回 `demo_reset_token` 等；已/未注册邮箱响应 `data` 结构差异（**FR 1.5** 高风险）。 |
| TI-F-07 | 密码重置提交 | `POST /api/reset` | 令牌校验、过期、`used`、新密码策略与确认密码（5001–5005 等）。 |

### 2.2 目标应用主要非功能特性（Non-Functional）

对齐 *Assignment 2* §1.1.2（Performance, Usability, Security, Maintainability and Technology），结合本应用现状：

| 测试项 ID | 类别 | 关注点 | 与实现的对应 |
|-----------|------|--------|--------------|
| TI-N-01 | **Security** | 会话密钥、Cookie、令牌、调试模式 | `LOGIN_APP_SECRET` / 默认 `SECRET_KEY`；`SESSION_COOKIE_HTTPONLY`、`SameSite`；`debug=True`；重置令牌在 JSON 中返回（风险报告 **FR 1.5、NFR 1.1.2**）。 |
| TI-N-02 | **Security** | 传输与 Cookie 标志 | 默认 HTTP 演示；`config.py` 未设置 `SESSION_COOKIE_SECURE`（若仅本地 127.0.0.1 可文档声明假设）。 |
| TI-N-03 | **Usability (UX/UI)** | 表单与错误提示 | 中文 `message`；`profile.html` 以 JSON 展示资料对非技术用户的友好性（记录为观察项）。 |
| TI-N-04 | **Performance** | SQLite 与并发 | 可选：多请求顺序/轻量并发冒烟，**不设**未经测量的 SLA 数字。 |
| TI-N-05 | **Maintainability / Technology** | 依赖与可重复环境 | `requirements.txt`（Flask、Flask-SQLAlchemy、bcrypt）；Python 3.9+；`app.db` 本地初始化。 |

### 2.3 系统架构与主要组件（简要）

| 层次 | 组件 | 职责 |
|------|------|------|
| **表示层** | `static/*.html`、`style.css` | 用户输入、调用 `fetch` 访问 `/api/*`，`credentials: "same-origin"`（以页面脚本为准）。 |
| **应用层** | `app.py`（Flask） | 路由、会话、业务编排、统一 JSON 错误处理（含 `code=9999` 服务器错误路径）。 |
| **领域与校验** | `services.py` | 输入校验、bcrypt、锁定时间计算。 |
| **持久层** | `models.py` + SQLite `app.db` | `User`、`PasswordResetToken` 表。 |
| **配置** | `config.py` | 数据库 URI、`SECRET_KEY`、会话生命周期、Cookie 策略。 |

## 三、基于风险的高层测试套件设计（High-level Test Suite Design）

设计原则（与 *Assignment 2* 及《风险分析报告》一致）：**先满足高风险（High）区域**，再覆盖 Medium，最后 Low；测试技术对齐 **ISO 29119-4** 常见黑盒方法及 **状态转换**（与作业 **FR 3.0、FR 4.0** 及「Mainly」流程中的策略—用例思路一致）。

### 3.1 风险与套件映射总览

| 风险报告优先级 | 涉及测试项（节选） | 套件名称（建议） | 主要测试技术 |
|----------------|-------------------|------------------|--------------|
| **High** | TI-F-02、TI-F-03、TI-F-06、TI-N-01 | **套件 A：认证与账户安全** | EP、BVA、**状态转换**、安全清单（密钥/令牌/Cookie） |
| **High** | TI-F-06、TI-F-07、TI-N-01 | **套件 B：密码重置与令牌** | EP、BVA、**决策表**（令牌有效/过期/已用 × 密码合法与否） |
| **Medium** | TI-F-01、TI-F-04、TI-N-03 | **套件 C：注册与会话资料** | EP、BVA、场景测试 |
| **Medium** | TI-N-04、TI-N-05 | **套件 D：可靠性与环境**（可选） | 轻量并发或顺序压力、环境检查清单 |
| **Low** | TI-F-05 | **套件 E：登出与残余状态** | 场景 + 探索性测试 |

### 3.2 套件 A：认证与账户安全（高风险优先）

**风险依据**：FR 1.1、FR 1.2、NFR 1.1.2（《风险分析报告》§4，Score ≥ 15 或锁定逻辑 High）。

| 设计要点 | 内容 |
|----------|------|
| **等价类划分（EP）** | 合法/非法用户名；合法/非法密码；已锁定 / 未锁定账户；正确/错误密码。 |
| **边界值分析（BVA）** | 失败次数 4→5 边界；锁定后第 1 次登录尝试；用户名长度 3、20 及越界。 |
| **状态转换（FR 4.0）** | 未登录 → 失败累计 → 锁定 →（时间或数据恢复）→ 未登录；未登录 → 已登录 → 登出。非法迁移：锁定下用正确密码仍拒绝。 |
| **安全清单** | 默认 `SECRET_KEY` 与设置 `LOGIN_APP_SECRET` 后的对比（若测试环境允许）；`Set-Cookie` 是否 `HttpOnly`；`debug` 是否对外暴露的说明性检查。 |

**套件目标**：证明登录与锁定主路径及高风险分支可验证、与风险登记一致；为工具导出的覆盖项提供执行证据。

### 3.3 套件 B：密码重置与令牌（高风险优先）

**风险依据**：FR 1.5（演示令牌泄露、枚举线索）、FR 1.6（Medium 但依赖令牌链）。

| 设计要点 | 内容 |
|----------|------|
| **EP** | 已注册邮箱 / 未注册邮箱 / 非法邮箱；令牌空/错/过期/已使用。 |
| **BVA** | 新密码长度与复杂度边界与注册一致；令牌过期时间前后（可配合改库或受控环境）。 |
| **决策表** | 条件：令牌有效、密码合法、两次密码一致 → 预期成功；各取反组合 → 对应 `code` 与 HTTP 状态。 |

**套件目标**：固定「演示行为」基线，便于答辩说明与生产差异；支撑详细设计文档中的追溯表。

### 3.4 套件 C：注册与会话资料（中风险常规）

**风险依据**：FR 1.0、FR 1.3（Medium）。

| 设计要点 | 内容 |
|----------|------|
| **EP / BVA** | 注册字段各等价类与边界；`GET /api/profile` 在登录前后、用户删除后（若手工准备数据）。 |
| **场景** | 注册后立即登录再访问资料；重复注册同用户名。 |

### 3.5 套件 D、E（可选与低优先级）

- **套件 D**：SQLite 轻量冒烟、依赖安装与端口冲突——保障**可测性**（项目风险 PR 1.0 缓解）。  
- **套件 E**：登出后立即与延迟访问资料、多标签行为——**Low** 深度、探索性为主。

### 3.6 与 AutoTestDesign 工具的衔接（说明性）

测试用例与覆盖项可在组内工具链中由 **`04_coverage_items.json`** 等产物驱动设计；**人类设计者**按作业 *Mainly* 要求对覆盖项、策略与用例做交互修订后，再映射到上述套件执行顺序。**本初稿不替代**工具内部 SCHEMA 或脚本实现细节。
