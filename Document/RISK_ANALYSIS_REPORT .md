# 目标应用风险分析报告：target-login-app（Web 登录系统）

## 1. 引言

### 1.1 报告目的

在课程期末被测场景下，本报告基于 **target-login-app**（Flask + SQLite）的**真实代码与接口行为**，梳理功能、可靠性、安全及文档不足类风险；以 **Likelihood × Impact** 映射测试优先级，支撑 **FR 2.0** 的资源分配，并与 **FR 3.0**、**FR 4.0** 及 **NFR 1.1.2** 对齐，输出可追溯结论，降低遗漏高风险路径的概率。

### 1.2 目标应用概述

| 维度 | 说明 |
|------|------|
| **功能** | 用户注册、登录与会话、登出、登录后个人资料、连续登录失败 **5 次锁定 30 分钟**、密码重置请求与提交；静态页 `index.html`、`login.html`、`register.html`、`profile.html`、`reset.html`。 |
| **目标用户** | 课程/期末项目中的**本地演示与验收**使用者；非生产用途，生产需配置 `LOGIN_APP_SECRET`。 |
| **核心业务流程** | 注册 → 登录 → 访问资料 → 可选登出；或「忘记密码」→ 请求重置 → 使用令牌设新密码 → 再登录。 |

**实现依据**：`app.py`、`services.py`、`config.py`、`models.py`、`README.md`、`static/*.html`。

## 2. 风险识别维度

### 2.1 产品质量风险（Product Risks）

| 子维度 | 与本应用相关的扫描结论 |
|--------|------------------------|
| **功能性** | 注册校验（用户名正则、密码复杂度、确认密码、邮箱格式）、登录错误码、锁定阈值、重置令牌过期与 `used` 标记等逻辑分支多，任一分支错误都会导致无法注册/无法登录/错误锁定或无法解锁/重置失败。 |
| **可靠性** | 使用 **SQLite**（`config.py` 中 `sqlite:///.../app.db`），多进程/高并发写入时可能出现锁竞争或性能抖动；`app.py` 主入口以 **`debug=True`** 运行，若误用于对外环境会放大异常暴露与服务不稳定风险。 |
| **安全性（NFR 1.1.2）** | 密码使用 **bcrypt** 存储；`SESSION_COOKIE_HTTPONLY`、`SameSite=Lax` 已配置；但 **`SECRET_KEY` 默认固定字符串**、**重置接口在 JSON 中返回演示用 `demo_reset_token`**、**个人资料返回邮箱**等均构成隐私或误部署场景下的安全关注点；POST API **未见 CSRF Token**。 |
| **可用性** | 各页为轻量表单 + `fetch`；`profile.html` 以原始 JSON 展示资料，对非技术用户友好性一般；错误信息为中文，与作业验收匹配。 |

### 2.2 项目风险（Project Risks）

| 风险点 | 说明 |
|--------|------|
| **文档不全** | `README.md` 以运行与 API 表为主，缺少正式需求规格（SRS）、威胁模型、非功能指标，易导致测试对「演示行为 vs 生产期望」边界理解不一致，从而覆盖不足或过度测试。 |
| **演示与生产混同** | `README` 与 `reset.html` 均说明「演示 token」；若团队未统一「仅本地演示」的约束，易在评审中产生错误安全假设。 |

## 3. 风险评估矩阵（FR 2.0）

### 3.1 评分标尺（1–5 分）

**发生可能性（Likelihood）**：缺陷或不利事件在典型使用/误用/攻击场景下出现的概率。

| 分数 | 含义 |
|------|------|
| **1** | 极难出现（需极端条件或几乎不会触发）。 |
| **2** | 较难出现（低频路径或需特定操作序列）。 |
| **3** | 中等（正常或常见操作中较易触发）。 |
| **4** | 较易出现（高频路径、弱约束或常见误配置下易发生）。 |
| **5** | 极易出现（默认配置、主路径或常见攻击面下几乎必然暴露）。 |

**影响程度（Impact）**：一旦发生，对用户、数据、业务或合规造成的损失大小（含安全与隐私，对齐 **NFR 1.1.2** 关切）。

| 分数 | 含义 |
|------|------|
| **1** | 可忽略：无实质影响或仅文案/观感问题。 |
| **2** | 轻微：个别用户短暂不便，无数据与安全后果。 |
| **3** | 一般：功能受阻或体验明显下降，或低风险信息暴露。 |
| **4** | 严重：核心功能错误、批量用户受影响，或中等敏感数据/会话面临威胁。 |
| **5** | 灾难性：系统级不可用、账户体系被突破、高敏感数据大规模泄露或合规不可接受。 |

### 3.2 风险评分与测试优先级（数值定义）

**风险评分**：取值范围为 **1–25**。本报告按下表将 **Risk Score** 映射为 **测试优先级**（与第 4 节登记表一致）：

| 测试优先级 | Risk Score 区间 | 测试策略含义 |
|------------|-----------------|--------------|
| **High** | **Score ≥ 15** | 核心路径、安全敏感或高影响缺陷；须**优先排期**并做**深度测试**（含负面场景、组合场景与专项）。 |
| **Medium** | **7 ≤ Score ≤ 14** | 重要但非最高爆发度；**常规测试**覆盖主流程与主要边界。 |
| **Low** | **Score ≤ 6** | 次要或影响面有限；在**资源允许**时测试，或以抽样/探索性方式覆盖。 |

**说明**：若某条风险 **Impact = 5** 且与身份、会话、令牌或密钥直接相关，即使 Score 落在 14，仍可在工具中手动标为 High；本登记表已按上表区间标注，与当前各条 **Likelihood × Impact** 结果一致。

## 5. 风险缓解策略与测试建议（FR 3.0 / FR 4.0）

本节将「**缓解**」与「**验证**」分开写，并尽量落到 **接口路径、字段与代码行为**，便于编写用例与自动化脚本（如 pytest + requests / Flask test_client）。

### 5.0 策略总览（与 FR 3.0 的对应关系）

| 测试技术 | 主要适用风险 | 在本应用中的典型产出 |
|----------|--------------|----------------------|
| **等价类划分（EP）** | FR 1.0、FR 1.1、FR 1.6、NFR 输入校验 | 合法/非法用户名、密码复杂度、邮箱、JSON 缺字段等价类 |
| **边界值分析（BVA）** | FR 1.0、FR 1.2、FR 1.6 | 长度上/下边界、失败次数 4/5/6、锁定截止时间前后 |
| **决策表** | FR 1.0、FR 1.6 | 多条件组合（校验项 × 确认密码一致与否） |
| **状态转换测试（FR 4.0）** | FR 1.1、FR 1.2、FR 1.3、FR 1.4 | 会话与锁定相关的合法/非法迁移 |
| **基于错误猜测 / 安全清单** | FR 1.5、NFR 1.1.2 | 令牌泄露、枚举、默认密钥、Cookie 属性 |

### 5.1 高风险项：缓解策略与详细测试建议

#### 5.1.1 FR 1.1 用户登录

**缓解策略**

- 部署与验收环境**必须**设置环境变量 `LOGIN_APP_SECRET`，避免沿用 `config.py` 中的默认 `SECRET_KEY`。
- 明确「仅本地演示」时仍建议在验收文档中写明：**禁止**将默认密钥部署到可被他人访问的网络。
- 若未来接入反向代理，约定 **HTTPS** 终止与 Cookie Secure 策略。

**测试建议（黑盒为主）**

| 序号 | 目的 | 建议步骤 / 数据 | 期望与记录要点 |
|------|------|-------------------|----------------|
| L1-1 | 主成功路径 | `POST /api/login`，合法 `username`、`password`，`Content-Type: application/json` | `code === 0`，响应含 `user_id`、`username`；浏览器或客户端带 Cookie 后续 `GET /api/profile` 成功 |
| L1-2 | 空输入 | `username` 或 `password` 为空串 / 缺字段 | `code === 2001`，HTTP 400（与当前实现一致） |
| L1-3 | 错误凭据（用户存在） | 错误密码 | `code === 2002`，HTTP **401**；`failed_attempts` 递增（可通过 DB 或后续锁定行为间接验证） |
| L1-4 | 错误凭据（用户不存在） | 不存在的用户名 + 任意密码 | 同样 `2002` / **401**；**验证数据库中无对应用户时不写入失败计数**（与 FR 1.2 联动） |
| L1-5 | 非 JSON / 非法 JSON | `get_json(silent=True)` 行为 | 空 body 时应不崩溃，走空字段校验路径 |

**白盒 / 配置类补充**：在测试中读取 `app.config["SECRET_KEY"]` 或环境变量，断言**非**默认 dev 字符串（适用于 CI 的 staging 配置）。

#### 5.1.2 FR 1.2 失败计数与账户锁定（5 次 / 30 分钟）

**缓解策略**

- **产品层**：若要求「任意用户名」均消耗锁定配额，需在实现上为**不存在用户**也维护计数或引入 **IP / 设备级限流**。
- **短期工程缓解**：在网关或 WAF 上对 `/api/login` 做速率限制，降低撞库与枚举成本。

**测试建议（EP + BVA + 状态转换 FR 4.0）**

**等价类**

- **已注册用户名 + 错误密码**：应累计失败，直至第 5 次后锁定。
- **未注册用户名**：每次均返回统一错误，且**不**应出现 `2003` 锁定码。

**边界值（失败次数）**

| 场景 | 连续错误密码次数 | 期望 HTTP / code |
|------|------------------|------------------|
| 第 4 次失败后 | 4 | 仍为 401 / 2002，**未**锁定 |
| 第 5 次失败后 | 5 | 401 / 2002，且此后**同用户**正确密码也应 **403 / 2003**（在 `locked_until` 有效期内） |
| 第 6 次（已锁定后） | — | 保持 403，直至模拟时间超过 `locked_until` 或手动改库 |

**状态转换图（文字版）**

- **S0 未登录（未锁定）** —错误密码→ **S1 未登录（失败 n 次，n 为 1～4）** —继续错误→ … —第 5 次→ **S2 未登录（已锁定）** —时间过期或人工清库→ **S0**；
- 从 **S0/S1** 一次正确密码→ **S3 已登录（会话有效）**。

**非法迁移用例**：在 **S2** 下提交正确密码，应仍被拒绝（锁定优先）；在 **S3** 下重复 login 行为（若允许）应不产生异常。

**观测点**：除 HTTP 外，可对 SQLite 中 `users.failed_attempts`、`users.locked_until` 做断言，或在接口层用「锁定后错误消息文案」做弱断言。

#### 5.1.3 FR 1.5 密码重置请求（演示令牌）

**缓解策略**

- **演示环境**：保留 `demo_reset_token` 时，限制访问 IP、或仅在无网络隔离的本地使用；文档中强调**禁止**截屏外泄响应 JSON。
- **类生产 / 验收**：应通过配置开关**关闭**响应中的令牌字段，改为仅服务端写日志或发邮件（需开发变更时评审）。

**测试建议**

| 序号 | 目的 | 建议步骤 | 期望 |
|------|------|----------|------|
| R5-1 | 已注册邮箱 | `POST /api/reset-request` 合法邮箱 | `code === 0`，`data` 中含 `demo_reset_token`、`demo_reset_link`，`expires_at` 为合理未来时间 |
| R5-2 | 未注册邮箱 | 格式合法但未注册邮箱 | `code === 0`，但 `data.demo` 提示无令牌；**记录响应结构差异**以评估枚举风险 |
| R5-3 | 非法邮箱 | 不符合 `validate_email` | 非 0 code（当前为校验失败路径） |
| R5-4 | 旧令牌失效 | 同一用户连续两次请求重置 | 验证仅**最新**未使用令牌有效 |

**安全回归**：确认生产构建中若引入「不返回 token」分支，自动化应对 `demo_reset_token` **不存在**仍能完成邮件/日志侧验收（若有）。

#### 5.1.4 NFR 1.1.2 安全与隐私（横切）

**缓解策略**

- 密钥、调试模式、数据库文件权限纳入**部署检查清单**；`README` 已提示 `LOGIN_APP_SECRET`，应在 CI 或启动脚本中**强制校验**。
- 对 CSRF：若未来嵌入第三方站点，需评估 `SameSite=Lax` 是否足够，必要时为状态改变接口增加 Token 或双重 Cookie。

**测试建议**

- **Cookie 属性**：登录后检查 `Set-Cookie` 是否含 `HttpOnly`；在浏览器开发者工具中确认前端脚本**不可读**会话 Cookie 名。
- **登出后会话**：`POST /api/logout` 后带原 Cookie 访问 `/api/profile`，应 401 / 3001。
- **异常处理**：对非 HTTP 异常路径触发内部错误，验证返回为统一 JSON `9999` 且**不**泄露堆栈到客户端。

### 5.2 中风险项：缓解策略与详细测试建议

#### 5.2.1 FR 1.0 用户注册

**缓解策略**：保持 `services.validate_*` 与前端提示一致；注册失败时**不**应创建半条用户记录。

**测试建议（EP / BVA / 决策表）**

- **用户名 BVA**：长度 2、3、20、21；字符 `-`、中文、空格；合法 `abc`、`a1_`。
- **密码 BVA**：长度 7、8、30、31；缺小写 / 缺大写 / 缺数字的**单缺陷**密码各一类。
- **确认密码**：与 `password` 一致 / 不一致两类。
- **邮箱**：无 `@`、`a@b`（无域）、合法 `a@b.co`。
- **唯一性**：同一用户名注册两次，第二次 `1005`。
- **决策表列**：用户名合法? × 密码合法? × 邮箱合法? × 两次密码一致? → 预期 `code` 与 HTTP 状态。

#### 5.2.2 FR 1.3 会话与个人资料

**测试建议**

- 未带会话：`GET /api/profile` → `3001`，HTTP **401**。
- 登录后：`data` 含 `id`、`username`、`email`、`created_at`；确认**无** `password_hash` 字段（与 `to_public_dict` 一致）。
- **用户删除与会话**：若手动删除 `users` 中对应行，再 `GET /api/profile`，应 `3002` 且会话被清除（与 `app.py` 行为一致）。

#### 5.2.3 FR 1.6 密码重置提交

| 类别 | 用例要点 |
|------|----------|
| 令牌 | 空 token、错误 token、已使用 token、过期 token（可改库 `expires_at` 或 mock 时间） |
| 密码策略 | 与注册相同的复杂度边界；`new_password` 与 `confirm_password` 不一致 → `5003` |
| 成功路径 | 合法 token + 合法新密码 → `code === 0`，随后用**新**密码登录成功、**旧**密码失败 |

#### 5.2.4 PR 1.0 文档不足导致的覆盖风险

**缓解策略**

- 建立「**最小需求说明**」一页纸：接口列表、错误码表、锁定规则、演示令牌语义、与生产差异。
- 每条高风险在工具中链接到**至少 3 条**可执行用例。

**测试建议**

- 做**需求-用例追溯矩阵**：行 = FR 1.0–1.6、NFR；列 = 用例 ID、技术（EP/BVA/状态图）、自动化与否、最后执行日期。
- 对 `README` 未写清的行为（如「用户不存在是否计数」），以**当前代码为准**写基线用例，并在需求变更时同步改期望。

### 5.3 低风险项：FR 1.4 登出

**缓解策略**：前端在登出成功后**立即**跳转登录页并清空页面上的敏感展示（当前 `profile.html` 以 JSON 展示，跳转可避免残留）。

**测试建议**

- `POST /api/logout` 后立刻 `GET /api/profile` → 未登录错误。
- **探索性**：登出后浏览器「后退」是否仍显示旧页面内容；再发请求应仍 401。
- **双会话 / 多标签**：单浏览器多标签登出是否相互影响。

### 5.4 FR 4.0 状态转换建模小结（便于画图与用例生成）

建议在作业或工具中单独附图，节点与迁移如下（文字版）：

1. **未登录** —正确登录→ **已登录**
2. **未登录** —错误密码（用户存在）→ **失败计数 +1**；若计数达 5 → **锁定中**
3. **锁定中** —时间未到期→ 拒绝登录（403）
4. **锁定中** —时间到期→ **未登录**（失败计数是否清零以产品为准；当前成功登录会清零失败与锁定）
5. **已登录** —登出→ **未登录**
6. **已登录** —会话过期 / 服务端重启密钥变化→ **未登录**

覆盖要求：**每条迁移至少 1 条用例**；每个**非法迁移**（如锁定中期望直接变已登录）至少 1 条负例。

## 6. 结论

**整体风险水平**：在「课程本地被测系统」前提下为 **中等偏高**；若按 `README` 仅本地使用，最大风险集中在 **误部署**（默认 `SECRET_KEY`、`debug=True`、SQLite）与 **密码重置演示设计**（令牌出现在 HTTP 响应体）上。

**测试重点聚焦区域**

1. **登录 + 锁定逻辑**，尤其「用户名不存在」路径不触发失败计数与产品安全预期是否一致；
2. **密码重置全链路**（令牌生成、过期、`used`、响应泄露与枚举线索）；
3. **会话与密钥配置**（`LOGIN_APP_SECRET`、Cookie、登出后会话状态）；
4. 在此基础上再展开注册校验边界与个人资料隐私字段的常规中风险测试。
